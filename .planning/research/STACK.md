# Technology Stack

**Project:** Libre Glucose MQTT Bridge — HA Supervisor app wrapping gluco-hub-rs
**Researched:** 2026-06-07
**Confidence:** HIGH (Context7 unavailable; verified via official HA docs + upstream gluco-hub-rs source + real project codebase)

## Recommended Stack

### Core Framework

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| gluco-hub-rs | current (CalVer) | Polls LibreLink Up, publishes to MQTT with HA discovery | Already in place; all resilience lives here — add-on wrapper is pass-through |
| Bash (bashio) | HA base image bundled | Entrypoint: reads options.json, exports GLUCO_HUB__* env vars | Required by HA Supervisor lifecycle; bashio is the only supported config/secret bridge |
| Docker (HA base image) | ghcr.io/home-assistant/base | Container runtime | Required by HA Supervisor; s6-overlay for init, bashio pre-installed |

### Config Transport

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| config.yaml (YAML) | HA Supervisor schema | User-facing options + validation | Required; HA Supervisor reads this for the app UI form |
| /data/options.json | Generated by Supervisor | Runtime config passed to container | Read by bashio::config in run.sh |
| GLUCO_HUB__* env vars | gluco-hub-rs config loader | Bridge between HA options and Rust binary | Zero-file config — no TOML generation needed; env vars override TOML key-by-key |

### MQTT Infrastructure

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Mosquitto add-on | HA bundled | MQTT broker | Required dependency; `mqtt:need` in services; bashio::services provides creds |
| rumqttc (via gluco-hub-rs) | 0.25 | MQTT v5 client | Upstream Rust crate; TLS (rustls), exponential reconnect, LWT |

### Supporting Libraries (upstream Rust)

| Library | Purpose | When to Use |
|---------|---------|-------------|
| validator (Rust crate) | Config validation at startup | Already in gluco-hub-rs; HA schema validates at UI level too |
| tracing (Rust crate) | Structured JSON logging | Already configured; RUST_LOG env var for verbosity |
| tokio (Rust crate) | Async runtime | Multi-threaded; poll loop + HTTP + MQTT eventloop |

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Multi-account approach | Multiple add-on instances OR upstream multi-source | Single process spawning multiple binaries | Multi-instance = cleaner: one process per account, independent failure domains, no new Rust code. Upstream multi-source = cleaner UX: one config panel. See Architecture decision below. |
| Config transport | GLUCO_HUB__* env vars | Generated TOML file | Env vars are the upstream design target; avoid file I/O in bashio wrapper |
| Resilience layer | Upstream Rust (DLQ + watermark + backoff) | Bash-level retry loop | Bash retry is fragile, duplicates upstream work; Rust binary already has production-grade resilience |
| MQTT client | rumqttc (via gluco-hub-rs) | mosquitto_pub shell calls | rumqttc provides LWT, retained messages, QoS, TLS — shell calls cannot match this |

## Key Architecture Decision: Multi-Account Strategy

**Problem:** gluco-hub-rs currently supports exactly one `[source.llu]` block — one LibreLink Up account, one patient. ENHN-01 requires multiple accounts.

**Option A — Multiple add-on instances (different slugs):**
- User installs the app multiple times under different slugs
- Each instance gets its own config, MQTT client_id, topic_prefix
- No Rust changes needed; pure config/UX work
- Cons: HA app store doesn't natively support multi-instance; user must manually set slug in app store URL
- Status: HA Supervisor does support this (Slug uniqueness is per-repo, not global)

**Option B — Upstream multi-source in gluco-hub-rs:**
- Change `SourceConfig.llu: Option<LluSourceConfig>` to `Vec<LluSourceConfig>`
- Multiple poll tasks or a single poll loop iterating accounts
- Each account gets its own topic_prefix, client_id, MQTT discovery
- Requires upstream Rust changes + HA config schema changes
- Pros: Single app instance, unified config panel

**Option C — Add-on wrapper spawns multiple gluco-hub processes:**
- run.sh iterates over a list config, exports per-account env vars, spawns N gluco-hub processes
- Each process has its own MQTT client (distinct client_id)
- Requires list-type config schema in HA + bashio iteration in run.sh
- Cons: Multiple Rust binaries = higher memory; process supervision gets complex
- Status: Feasible but fragile — s6-overlay expects one main process

**Recommendation:** Option A for v1.1 (fastest path, zero Rust changes). Plan Option B for a future milestone when upstream gluco-hub-rs adds multi-source support. Option C is a fallback if multi-instance UX proves unacceptable.

## HA Add-On Schema: List-Type Configuration Patterns

### How HA schema supports lists

The HA Supervisor schema (in `config.yaml`) supports YAML lists with typed entries:

```yaml
# Example: single account (current v1.0)
options:
  llu_email: ""
  llu_password: ""
  llu_region: "EU"

schema:
  llu_email: email
  llu_password: password
  llu_region: "list(AE|AP|AU|CA|DE|EU|EU2|FR|JP|US|LA|RU|CN)"
```

```yaml
# Example: multi-account with list schema (Option C / Option B)
options:
  accounts:
    - email: ""
      password: ""
      region: "EU"
      patient_id: ""
      timezone: "UTC"
      client_id: "ha-1"
      topic_prefix: "gluco-hub/ha-1"

schema:
  accounts:
    - email: email
      password: password
      region: "list(AE|AP|AU|CA|DE|EU|EU2|FR|JP|US|LA|RU|CN)"
      patient_id: "str?"
      timezone: "str"
      client_id: "match(^[A-Za-z0-9_-]{1,23}$)"
      topic_prefix: "str"
```

**Schema rules from HA docs** (source: developers.home-assistant.io/docs/apps/configuration):
- Lists can contain dictionaries with typed fields (max nesting depth 2)
- `?` suffix makes a field optional (e.g., `"str?"`)
- Supported types: `str`, `bool`, `int`, `float`, `email`, `url`, `password`, `port`, `match(REGEX)`, `list(val1|val2|...)`
- `str(min,max)` for length constraints
- `int(min,max)` for range constraints
- Default values set in `options:`; omit from `options:` + add `?` in `schema:` for truly optional

### Real add-on patterns observed

- **Mosquitto broker**: uses list schema for `customize` and `certs` sections
- **Zigbee2MQTT**: uses `devices:` list for per-device overrides
- **Node-RED**: uses `credential_secret` as a simple string, not list
- **Common pattern**: most HA add-ons with "multiple things" use separate add-on instances rather than list configs

### bashio iteration over lists

```bash
# Reading list items from bashio (for Option C)
ACCOUNT_COUNT=$(bashio::config 'accounts|length')
for i in $(seq 0 $((ACCOUNT_COUNT - 1))); do
    EMAIL=$(bashio::config "accounts[${i}].email")
    PASSWORD=$(bashio::config "accounts[${i}].password")
    # ... export per-account env vars
done
```

bashio uses `jq` internally — list indexing is zero-based. The `|length` filter returns array length.

## gluco-hub-rs Upstream: Current Multi-Account Support

### What exists today

**Single account only.** Confirmed via source code analysis:

```rust
// gluco-hub/src/config.rs
pub struct SourceConfig {
    pub llu: Option<LluSourceConfig>,  // SINGLE optional LLU source
}

pub struct LluSourceConfig {
    pub email: String,
    pub password: Option<SecretString>,
    pub password_file: Option<PathBuf>,
    pub region: String,
    pub patient_id: Option<String>,  // filters WITHIN one account
    pub version: Option<String>,
    pub timezone: Option<String>,
}
```

**Patient selection within an account:** `patient_id` picks one patient from the `/llu/connections` response. Multiple patients under one account are supported — the `ConnectionSelection` enum handles `First` (default) vs `ByPatientId(PatientId)`.

**Source architecture:** `build_default_source()` builds ONE source. The poll loop runs one `tokio::spawn` task. The `Source` trait is single-instance by design.

### What upstream would need for native multi-account

1. Change `SourceConfig.llu: Option<LluSourceConfig>` to `Vec<LluSourceConfig>`
2. Change `build_default_source` to return `Vec<BuiltSource>`
3. Spawn one poll task per source, each with its own MQTT sink config (client_id, topic_prefix)
4. Each account's MQTT sink publishes to its own `topic_prefix` with its own `client_id`
5. Each account's HA discovery messages go to `{discovery_prefix}/sensor/gluco_hub_{client_id}_{kind}/config`

**Current blocker:** Upstream is pre-1.0 beta. Multi-source is not in the roadmap visible in docs/ARCHITECTURE.md or config.example.toml. It would be a non-trivial refactor touching config.rs, main.rs (poll loop architecture), and the MQTT sink instantiation.

## API Resilience Patterns

### What gluco-hub-rs already provides (upstream, no wrapper needed)

**Source resilience (LibreLink Up):**

| Mechanism | Implementation | Trigger |
|-----------|---------------|---------|
| Token cache | `Arc<Mutex<Option<LluTokens>>>` in `LluSource` | Tokens reused across polls; shared re-login on expiry |
| Expiry skew | 60s before stated expiry | Avoids boundary-condition 401s |
| 401 retry | Single re-login + retry per data call | Token rejected → drop cache → fresh login → retry once |
| Error classification | `[LLU001]`–`[LLU009]` stable codes | Metrics, logs, dry-run exit codes |

**Sink resilience (MQTT):**

| Mechanism | Implementation | Trigger |
|-----------|---------------|---------|
| Per-sink watermark | `SinkRouter` — `last_pushed_ts` per sink | Drops already-pushed readings (288→1 per cycle steady state) |
| Watermark backfill | On recovery, watermark hasn't advanced → full window replayed | Sink offline for N cycles → next push replays N missed readings |
| Persistent DLQ | `DlqSink` — JSONL per sink at `state.dir/dlq/` | Push failure → write to disk; next success → drain + delete |
| DLQ eviction | FIFO eviction at `max_entries` (default 10K, ~35 days) | Queue full → oldest dropped, `cgm_dlq_evicted_total` increments |
| MQTT exponential backoff | 1s → 2s → 4s … → 60s cap in `run_poll_loop` | `EventLoop::poll()` error → backoff sleep before retry |
| LWT (Last Will) | `{"online":false}` retained on `_health` topic | TCP disconnect → broker publishes LWT automatically |
| Health re-publish | `{"online":true}` on every ConnAck | Reconnect → HA availability flips back |
| Stats snapshot | Periodic `_stats` topic (retained, configurable interval) | Independent task; survives push failures |
| Offline drain | Best-effort `online: false` publish on shutdown | SIGTERM → publish + drain PubAck before exit |
| Graceful shutdown | `CancellationToken` → poll loop exits cleanly | SIGTERM/SIGINT → cancel token |

**Startup validation:**

| Check | When | Failure mode |
|-------|------|-------------|
| Feature/config match | `verify_features()` | `[CFG006]` — TOML block references un-compiled feature |
| Secret presence | `verify_secrets()` | `[CFG007]` — secret env var empty |
| Config validation | `check-config` subcommand | `[CFG001]`/`[CFG002]` — missing/invalid fields |

**Liveness:**

| Signal | Mechanism | Consumer |
|--------|-----------|----------|
| Heartbeat file | `state.dir/.alive` touched after each poll tick | External watchdog (Docker HEALTHCHECK) |
| HTTP health | `GET /healthz` → `{"status":"ok"}` | HA Supervisor, monitoring |
| MQTT health | `_health` topic (retained + LWT) | HA availability_template |

### What the add-on wrapper (Bash/bashio layer) needs to add

**For ENHN-01 (multi-account):** Resilience is per-account, not per-wrapper. The wrapper's job is only config translation. No new resilience patterns needed at the Bash level — each gluco-hub process (or each account iteration within a multi-source binary) inherits all upstream resilience mechanisms.

**For ENHN-04 (API changes):** Upstream gluco-hub-rs already handles:
- Unknown LLU fields: serde ignores them by default (the `wire.rs` types use `#[serde(rename_all = "camelCase")]` with only expected fields)
- Missing optional fields: `Option<T>` deserializes to `None`
- Version header override: `GLUCO_HUB__SOURCE__LLU__VERSION` env var → no recompile needed
- Error classification: Stable `[LLU00x]` codes survive API shape changes

The wrapper should expose the `llu_version` config option (currently not in config.yaml schema) so users can override the LLU app version header without waiting for a gluco-hub-rs release.

**bashio has no built-in retry:** The bashio library does not provide retry/backoff functions. Any wrapper-level resilience (health check before starting, wait-for-MQTT loop) would need manual Bash implementation:

```bash
# Manual MQTT connectivity check (if needed — MQTT is a service dependency already)
for i in $(seq 1 30); do
    if nc -z "${MQTT_HOST}" "${MQTT_PORT}"; then
        break
    fi
    bashio::log.info "Waiting for MQTT broker..."
    sleep 2
done
```

However, this is unnecessary: `mqtt:need` in `services` already gates startup, and gluco-hub-rs's own MQTT backoff handles transient broker unavailability at runtime.

## MQTT Discovery for Multiple Sensor Entities

### Current single-account discovery (v1.0)

gluco-hub-rs publishes two retained discovery messages per ConnAck:

```
Topic: homeassistant/sensor/gluco_hub_{client_id}_glucose/config
Payload: {
  "name": "Glucose",
  "unique_id": "gluco_hub_{client_id}_glucose",
  "state_topic": "{topic_prefix}/glucose",
  "value_template": "{{ value_json.mgdl }}",
  "unit_of_measurement": "mg/dL",
  "state_class": "measurement",
  "device": {
    "identifiers": ["gluco_hub_{client_id}"],
    "name": "Gluco Hub ({client_id})"
  },
  "availability_topic": "{topic_prefix}/_health",
  ...
}

Topic: homeassistant/sensor/gluco_hub_{client_id}_trend/config
Payload: {
  "name": "Trend",
  "unique_id": "gluco_hub_{client_id}_trend",
  "state_topic": "{topic_prefix}/glucose",
  "value_template": "{{ value_json.trend }}",
  "device_class": "enum",
  "options": ["DoubleUp", "SingleUp", "FortyFiveUp", "Flat", ...],
  "device": { ... same device block ... },
  ...
}
```

### Multi-account discovery requirements

Each LibreLink Up account needs:

1. **Distinct `client_id`** — MQTT requires unique client IDs per connection. Current schema limits to 23 chars `[A-Za-z0-9_-]`. For multi-account: `ha-account1`, `ha-account2`, etc.

2. **Distinct `topic_prefix`** — Readings from different accounts must go to different topics so HA entities don't collide. Pattern: `gluco-hub/{client_id}` (already the default).

3. **Distinct `unique_id`** — HA discovery must have globally unique IDs. Current pattern `gluco_hub_{client_id}_{kind}` already handles this — different `client_id` → different `unique_id`.

4. **Distinct `device` identifiers** — Each account's entities group under its own HA device. Pattern: `["gluco_hub_{client_id}"]` works as-is.

5. **Distinct `discovery_topic`** — Each entity publishes to `{discovery_prefix}/sensor/{unique_id}/config`. Different `unique_id` → different topic.

6. **Distinct `availability_topic`** — Each account gets its own `_health` topic: `{topic_prefix}/_health`. Independent availability tracking.

### HA MQTT discovery format (reference)

From home-assistant.io/integrations/mqtt/#mqtt-discovery (HA 2024.6+):

- Discovery messages are **retained** with **QoS ≥ 1** on `{discovery_prefix}/sensor/{object_id}/config`
- HA subscribes to `{discovery_prefix}/#` on startup and after birth message
- Each entity gets a **unique_id** within the `device` context
- The `device.identifiers` array groups entities under one HA device
- The `origin` block (required for device discovery, HA 2024.6+) surfaces integration metadata
- `availability_topic` with `payload_available`/`payload_not_available` for LWT-driven availability
- `expire_after` can fall back if no LWT (not recommended for retained health topics)

### Multi-account discovery topology

```
Account 1 (client_id: ha-alice, topic_prefix: gluco-hub/ha-alice):
  → sensor.gluco_hub_ha_alice_glucose  (state_topic: gluco-hub/ha-alice/glucose)
  → sensor.gluco_hub_ha_alice_trend    (state_topic: gluco-hub/ha-alice/glucose)
  → device: Gluco Hub (ha-alice)
  → availability: gluco-hub/ha-alice/_health

Account 2 (client_id: ha-bob, topic_prefix: gluco-hub/ha-bob):
  → sensor.gluco_hub_ha_bob_glucose    (state_topic: gluco-hub/ha-bob/glucose)
  → sensor.gluco_hub_ha_bob_trend      (state_topic: gluco-hub/ha-bob/glucose)
  → device: Gluco Hub (ha-bob)
  → availability: gluco-hub/ha-bob/_health
```

No MQTT topic collision because `topic_prefix` differs. No unique_id collision because `client_id` differs. HA groups entities correctly because `device.identifiers` differ.
