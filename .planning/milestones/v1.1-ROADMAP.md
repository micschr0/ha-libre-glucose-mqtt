# Roadmap: Libre Glucose MQTT Bridge

## Overview

v1.1 enhances the bridge from single-account reliability to multi-account robustness. Phase 5 shores up API resilience — exposing version controls, hardening field parsing, adding rate-limit backoff — so the bridge survives LibreLink Up API changes without data loss or operator panic. Phase 6 extends gluco-hub-rs to multi-source polling, creates per-patient MQTT entities with guaranteed unique identity, and keeps existing single-patient configs working identically. Each phase delivers independently testable value to users: Phase 5 makes every deployment more resilient; Phase 6 unlocks family/household monitoring from one add-on instance.

### Phase 5: API Resilience

**Goal**: Bridge withstands LibreLink Up API changes — version header bumps, HTTP 429 rate limiting, field renames — without data loss or operator emergency
**Depends on**: Phase 4 (v1.0 complete)
**Requirements**: ENH-05, ENH-06, ENH-07, ENH-08, ENH-09
**Success Criteria** (what must be TRUE):

  1. User can override the LLU app version header (`llu_version`) from add-on config without waiting for an upstream gluco-hub-rs release
  2. Bridge handles HTTP 429 rate-limit responses with Retry-After backoff; all polled accounts resume after the rate-limit window without operator intervention
  3. Bridge logs a schema fingerprint at startup (hash of expected LLU JSON field names) so operators can detect LibreLink Up API changes before they cause silent data gaps
  4. Critical LLU JSON fields (glucose value, trend arrow, timestamp) tolerate renames or missing values without crashing the poll loop or producing zero-value readings
  5. Existing single-patient v1.0 configurations continue working identically — same MQTT topics, same entity IDs, same behavior (backward compatibility preserved)

**Plans**: TBD

### Phase 6: Multi-Patient Support

**Goal**: One add-on instance monitors multiple LibreLink Up accounts or patients, each with isolated MQTT entities, independent error recovery, and guaranteed collision-free discovery
**Depends on**: Phase 5 (API resilience)
**Requirements**: ENH-01, ENH-02, ENH-03, ENH-04, ENH-10
**Success Criteria** (what must be TRUE):

  1. User can configure multiple LibreLink Up accounts (or multiple patients within one account) from a single add-on config panel
  2. Each patient gets separate MQTT sensor entities (glucose + trend) with globally distinct `unique_id` — no HA discovery collisions even with multiple accounts
  3. Home Assistant auto-discovers glucose and trend sensors per patient as separate devices, each carrying the patient's configured display name
  4. One account's login failure, region mismatch, or LLU API error does not block other accounts from polling and publishing readings
  5. Existing single-account v1.0 configurations continue working identically — the first account retains legacy `client_id` and `topic_prefix` defaults, no entity renaming on upgrade
